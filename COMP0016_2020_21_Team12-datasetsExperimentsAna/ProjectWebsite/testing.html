Testing Strategy

For each component of our solution, we used different testing approaches. For instance, we used mainly manual testing for the data pipeline, whereas for the data visualization we used automated unit tests. In addition, we conducted usability tests on users from various backgrounds.

Data Pipeline

The ML SVM prediction model was manually tested by visualising the results produced. In addition, its performance was compared to the Linear Regression model by comparing the mean absolute error and the R2 score.

Progressive Web App

To ensure that the progressive web app behaves as expected, we first started writing unit tests. For the JavaScript code, we used the Jest universal testing platform. All the test files were written in the test folder of our PWA, and they can be executed using the "npm test" command.

To ensure that the progressive web app behaves as expected, we first started writing unit tests. For the JavaScript code, we used the Jest universal testing platform. All the test files were written in the test folder of our PWA, and they can be executed using the "npm test" command. Additionally, our build pipeline implemented in the Azure DevOps project will run the test and report any failure before producing the build artifact.

For the Azure Maps visualization, the code was written following the "separation of concern" design pattern. Thus, testing the application logic did not require mocking the functionalities provided by our API nor by the Azure Maps JavaScript library. For instance, we test that only the unique FAD ids are added to the map by providing a mock JSON object instead of the full dataset. By mocking the data for a FAD, we check the validity of the HTML created for its additional details. Similarly, we check the options for map objects such as lines.

In order to test the searching elements section of our PWA (at the bottom of the UI which you can select particular elements to search), we firstly ensure that the list for indicating the selected searching elements can be changed correctly. We use a list to pass required searching elements to the searching result page. Everytime the user selects an element, the corresponding keyword will be pushed to the list (A second click will remove it of course), then the data views page knows that it should display those particular data. By trying to push different elements to the list respectively and check the returned list, we can ensure that the list performs correctly. Secondly, we need to make sure that the information passed to the data views pages are correct. We use the URL ( html file name + ? + the list ) to tell the data views page which part of data should be displayed. By trying to assign different lists to the URL, and check the returned URL string, we can know whether the data views page can get the correct instructions. Then it comes to the API’s part.

API

To test the API, we used the request Python Library, which allows sending HTTP requests. For instance, by sending GET requests to the API, we tested that either the correct data was retrieved for a particular FAD id, or the 404 status code was sent if no data was found. Similarly, by sending PUT requests, we tested that the new FAD data is indeed added to the dataset.

User Acceptance Testing

We went through the codes and functionalities with our project partners during the last few weeks. Generally, we tested the core functionalities, with related code explained to them. We followed the overall pipeline of our project (Upload -> Store -> View) to see the perfomance of the product. Our project partners said that most parts are good, and they gave us some suggestions.

Firstly, the Azure account information (such as Access Keys) should be provided by the users when they deploying our project. We should not contain any information of our own Azure accounts, which is inappropriate and unsafe.

Secondly, instead of using the “magical number” (We originally used a list with four int, initialised to [0, 0, 0, 0] to pass searching elements. If a particular element is selected, we change the integer with corresponding index to 1.), we could use keywords in the list to tell the result page about the searching elements selected. We can initialise an empty list, when an element is selected, we can just push that keyword to the list. The benefit of doing this is that this approach is not only friendly for the developer since it clearly shows the information, but also for the actual users because they can understand the searching elements by their URL. Furthermore, they can even directly type in the elements they want to search follow the URL without operating them on the home page.

We also met our potential clients from Invictus FAD Team from Saint Lucia. We did a presentation to them, to demonstrate our implemented functionalities. They said that this is exactly what they want, and they appreciated for our hard works.
